# HLD — 06: Trade-offs

## Decision Log

| # | Decision | Chosen Approach | Rejected Alternative | Reason |
|---|---|---|---|---|
| 1 | Driver location store | **Redis GEOADD** (in-memory geo-index) | Postgres + PostGIS | Dispatch SLO is p95 ≤ 1 s. PostGIS geo-query on 300k rows takes 5–50 ms under load + network hop. Redis geo-query is O(log N + M) at < 1 ms. |
| 2 | Dispatch matching | **Pull candidates from Redis, rank in-service** | Dedicated geo-service with its own DB | Fewer network hops. Redis already holds the authoritative location data. Adding a geo-service adds a synchronous RPC on the critical dispatch path. |
| 3 | Multi-region DB | **CockroachDB active–active** | Postgres with read replicas + manual failover | CockroachDB provides region-pinned leaseholders and survives region failure automatically. Postgres replicas require manual promotion and risk split-brain. |
| 4 | Location events | **Kafka async** (published from Location Ingestion) | Direct gRPC push to Dispatch | 500 k/s cannot be handled synchronously end-to-end. Kafka decouples producers from consumers, provides backpressure, and enables replay. |
| 5 | Surge pricing consistency | **Eventual** (5 s compute cycle, 60 s Redis TTL) | Strong per-request computation | Surge is a pricing heuristic, not a financial record. Stale by ≤ 5 s is acceptable. Strong consistency would require a synchronous DB read on every ride request, violating the 1 s dispatch SLO. |
| 6 | Payment execution | **Async** (Kafka `trip.ended` triggers payment) | Synchronous in `POST /trips/{id}/end` | PSP latency is external and uncontrolled (can be 100 ms–10 s). Blocking the trip-end API response on PSP would degrade rider experience and violate internal SLOs. |
| 7 | Idempotency enforcement | **DB unique constraint** on `idempotency_key` + Redis NX dedup for in-flight | Application-level check only | DB constraint is the ultimate source of truth — survives restarts and concurrent races. Redis NX dedup prevents duplicate PSP calls before the DB write commits. |
| 8 | Geo-cell definition | **Uber H3 hex grid (resolution 8)** | Square grid (geohash) | H3 cells at res-8 (~460 m edge) are uniform in size with no polar distortion. Hexagons have consistent neighbour distance. H3 kRing() gives the N nearest cells with a single call — ideal for radius expansion when local supply is low. |
| 9 | Offer timeout | **10 s per offer, 30 s total** | Shorter/longer | 10 s gives driver enough time to react to a push notification. 30 s total keeps end-to-end p95 ≤ 3 s achievable when a match is found quickly; and degrades gracefully otherwise. |
| 10 | Surge multiplier locking | **Locked at match confirmation** | Computed at trip end | Locking at match confirmation gives the rider a predictable fare estimate from the moment a driver is assigned. Computing at end would allow surge to change mid-trip — unfair to the rider and harder to dispute. |
| 11 | Payment writes to `trips.payment_status` | **Trip Service consumes `payment.completed` event and updates its own row** | Payment Service writes directly to `trips` table | Giving Payment Service write access to Trip's schema violates service ownership. The clean approach is for Trip Service to subscribe to `payment.completed` / `payment.failed` Kafka events and update `payment_status` itself. This keeps each service the sole writer of its own data. |
| 12 | PII at rest | **Column-level AES-256 encryption** on `rider_name`, `driver_name`, `pickup_address`, `dropoff_address`, `phone_number` in CockroachDB; encryption key managed by cloud KMS | Encrypt entire DB volume only | Column-level allows query on non-sensitive columns without decryption overhead; satisfies GDPR/DPDP right-to-erasure (delete key = data inaccessible). |
| 13 | Payment card data (PCI DSS) | **PSP tokenisation — raw card data never touches our storage**. Card entry is handled by the PSP’s hosted fields; we receive only a PSP token and reference ID | Store encrypted card numbers ourselves | PCI DSS scope is reduced to SAQ A once card data leaves our domain entirely. |
| 14 | GDPR / DPDP right to erasure | **Pseudonymisation**: replace PII columns with a deterministic tombstone (`[DELETED]`) on erasure request; trip financial records are retained for 7 years (legal obligation) with PII removed | Full deletion of rows | Retaining financial records satisfies tax / dispute requirements; removing PII satisfies erasure rights. |
